# Agent 智能边界与能力演进规范

## 文档目标与适用范围
本文档把项目对“Agent 智能来自模型自主涌现，而不是主程序硬编码编排”的长期立场，收敛为可执行的规范约束。

本文档适用于所有会影响 Agent 决策空间、能力扩展路径、审批治理、skill 沉淀方式、tool surface 粒度、prompt 表达方式和站点经验落点的设计与实现工作。

本文档记录的是长期稳定的产品与架构约束，不是某一阶段的实现现状、实验方案、复盘或 memory dump。若实现与本文档冲突，应优先修正实现，或先更新本文档后再继续变更。

## 与现有规范的关系
本文档与 [`2026-04-20-agent-product-design-principles.md`](./2026-04-20-agent-product-design-principles.md) 的关系如下：
- 后者定义 Agent 产品层面的上位设计立场。
- 本文档进一步把“不要替 Agent 做决策”细化为能力设计、审批治理、skill 沉淀和缺口修复路径上的约束。

本文档与 [`2026-04-20-autonomous-agent-runtime-constraints.md`](./2026-04-20-autonomous-agent-runtime-constraints.md) 的关系如下：
- 后者约束运行时生命周期、prompt 落点、Global Memory 和站点接入边界。
- 本文档补充说明：当 Agent 能力不足时，主程序应该如何扩展环境与治理边界，而不回退成硬编码工作流。

## 核心立场
Agent 的智能，默认应来自 LLM 在目标、约束、工具和上下文上的自主理解与策略生成。

主程序的职责，是提供让这种智能得以发挥的环境，包括 prompt、memory、tool surface、plugin、skill 机制、审批治理、持久化和通用基础设施；主程序不应替 Agent 预写业务判断、站点操作剧本或失败兜底流程。

任何新增实现，都必须先回答一个问题：这项改动是在给 Agent 更多可用环境，还是在替 Agent 提前做决定。前者通常是正确方向，后者通常意味着系统正在向硬编码流程退化。

## 主程序与 Agent 的职责边界
| 层级 | 应负责 | 不应负责 |
| --- | --- | --- |
| 主程序 | 提供通用工具原语、结构化上下文、prompt 资产、治理规则、持久化与审计能力 | 替 Agent 编排固定步骤、预设站点剧本、预写单站点 fallback |
| Agent | 基于目标与世界状态自主决定下一步动作，组合工具执行任务，并在运行中沉淀 skill | 依赖主程序预置“某站点如何操作”的专属答案 |
| 治理层 | 定义哪些动作需要审批、暂停、handoff、恢复与审计，并在运行时强制执行 | 把安全性寄托于模型自觉判断，或把审批写成流程节点硬编码 |

## 禁止回退为工作流编排的形态
### 1. 禁止在 core runtime 中写死业务状态机
不得在 `runtime/`、`services/`、`plugins/` 或其它核心路径中，写死“先做 A，再做 B，失败则做 C”一类业务状态机或阶段枚举分支。

凡是把 Agent 的决策空间压缩成固定流程图的设计，都应视为对智能涌现原则的破坏。

### 2. 禁止把站点操作手册写进 prompt 或代码
不得在 prompt、工具描述、服务字符串或运行时代码中，写入某个站点的逐步操作手册、selector、固定按钮文案、页面专属路径或其它等价脚本。

允许提供的，只能是抽象层面的任务目标、约束、完成标准和通用页面语义提示，例如“招聘站点通常存在列表页与详情页”。

### 3. 禁止把单站点经验预写成主程序内置 skill
不得在仓库中预置“某站点一键同步”“某站点一键沟通”之类本应由 Agent 在运行中学习得到的 skill 成果。

主程序可以提供 `skill.create`、`skill.invoke`、`skill.list` 等机制，但 skill 的具体内容、步骤和策略应由 Agent 在真实运行中产出，并作为其学习资产持久化。

### 4. 禁止把能力缺口修复成站点专属 fallback
当外部站点页面变化、字段缺失、反爬干扰或临时链路失败时，主程序不得通过补写站点专属 fallback、下载补偿逻辑、字段猜测逻辑或页面专属重试分支来兜底。

允许进入主程序的，只能是与站点无关的通用能力改进，例如工具暴露更清晰、审批状态机更健全、结构化上下文更完整或持久化更可靠。

### 5. 禁止把流程封装成伪原子工具
工具应提供语义清晰、边界明确、可组合的原子能力，例如结构化读取、结构化写入、通用浏览器动作、审批恢复与状态查询。

不得把“运行某个完整招聘流程”“按既定步骤完成某站点任务”这类工作流封装成单个黑盒工具，再让 Agent 只是触发它。

### 6. 主 Agent 历史自然语言必须保持业务级抽象
主 Agent 历史层中的自然语言，本身就应保持业务级抽象，只表达目标、对象、进展、风险、阻塞和下一步等业务语义，而不应泄露浏览器标签页、URL、DOM 结构、页面按钮文案、切换页面、点击路径或其它网页执行细节。

浏览器标签页、页面跳转、元素定位、按钮点击、局部 DOM 观察、链路重试和同类网页执行细节，属于 subagent、子上下文或工具执行链路内部状态。它们应在内部被封装、提炼并转换成业务语义后，才允许上浮到主 Agent 历史层；不得把工具执行轨迹直接冒充成主 Agent 的业务历史。

若某条历史信息无法在脱离网页执行细节的前提下表达其业务意义，优先说明业务对象、当前状态和阻塞原因，而不是把页面过程原样抄写进主 Agent 历史。主 Agent 历史服务的是长期业务上下文与跨 run 理解，不是网页操作录像或调试日志。

### 7. 禁止在投影层改写 Agent 历史自然语言
本次越界的根因，是把系统应承担的结构化兼容责任，与 Agent 应承担的自然语言产出责任混淆了。系统负责修正结构化来源、字段语义和展示投影；Agent 负责在当时上下文中产出自然语言。前者不能通过改写历史文本去弥补后者，后者也不能把内部网页执行细节直接上浮到主 Agent 历史中，指望投影层事后再替它“修文案”。

`runtime`、`API`、`UI` 以及其它 projection / rendering layer，不得通过字符串重写、文案替换、规则匹配、正则归一化或其它同类手段，改写 Agent 已经产出的历史自然语言，以制造“当前规范口径”、补齐事后定义，或伪造看似一致的历史表达。

正确做法是优先修复结构化来源，并基于结构化字段重新生成展示。例如修正事件 schema、状态字段、对象引用、摘要字段、兼容映射或投影逻辑，然后让展示层从结构化数据重建标题、状态、标签、摘要或告警内容。

如果历史记录缺乏足够结构，允许展示层降级展示状态、标题、时间、对象引用或其它可验证的结构化片段，但不得改写原始自然语言文本本身。缺少结构时，宁可少展示，也不要通过猜测、替换或归一化去伪造一段“更符合当前规范”的历史文案。

这条约束适用于所有展示链路，包括但不限于 `memory projection`、`workspace summary`、`run / goal preview`、`dashboard alert`、历史记录渲染，以及其它任何把 Agent 历史内容投影给 human 或系统下游模块的路径。

## 能力缺口的正确修复顺序
当 Agent 在真实环境中表现不足时，默认应按以下顺序排查和修复：

1. 先检查目标表达、完成标准、角色边界和场景约束是否在 prompt 资产中表达清楚。
2. 再检查 tool surface 是否提供了足够清晰的原子能力，以及运行时是否透传了足够的结构化上下文。
3. 再检查 memory / skill 机制是否把可复用经验沉淀在了正确层级，而不是把瞬时执行细节混进长期资产。
4. 再检查审批、暂停、handoff、恢复和审计等治理能力是否足够完整，避免 Agent 因制度缺口被迫失败。
5. 只有当问题明确属于通用运行时缺陷时，才修改 core runtime；单一站点、单一页面、单一下载链路的问题，默认不得以 core 补丁收敛。

## Skill 资产治理
### 1. Skill 是 Agent 的长期学习资产
Agent 在运行中产出的 skill，必须作为长期学习资产持久化，而不是作为 run 级临时内存随执行结束而丢失。

SkillRegistry 必须支持跨 run、跨 session 的持续复用，否则“越跑越聪明”无法成立。

### 2. Skill 归属必须按 Agent 身份隔离
不同 Agent 角色沉淀的 skill 必须有明确归属，不得把 Assistant 与 Autonomous 的 skill 自动混用。

隔离的是 skill 的归属与使用作用域，不是 skill 机制本身。机制应统一，资产归属应清晰。

### 3. 主程序只负责 skill 机制，不预写 skill 内容
主程序负责提供 skill 的创建、调用、列举、审计和持久化机制。

主程序不负责预写某站点的 skill 正文，不负责把业务策略提前固化成默认 skill，也不负责替 Agent 维护某个站点的专属经验库。

## 审批治理必须在基础设施层生效
### 1. 审批要求必须由工具元数据与治理策略声明
哪些动作需要审批，必须由工具元数据和治理策略在注册或装配阶段声明，例如 `requires_approval` 或更细粒度的 `approval_policy`。

Runtime 在工具调用时必须自动判断是否拦截并创建 OperatorInteraction，而不是依赖 Agent 自己记得调用某个“请求审批”的额外工具。

### 2. 禁止把审批门建立在模型自觉之上
以下形态均应视为不合格设计：
- 让 LLM 自己判断“这一步是不是高风险，所以我应该先问人”。
- 在 prompt 中只写“高风险动作要谨慎”，但没有系统级拦截与审计。
- 在业务流程代码里写死“到了某个步骤就触发审批”，从而把审批逻辑绑定到固定工作流节点。

### 3. 审批内容应结构化表达
审批展示给 human 的信息，默认应来自工具调用的结构化语义，例如 `action_kind`、目标对象、参数摘要和潜在影响，而不是要求 Agent 手写一段自由文本解释自己想做什么。

治理层应保证审批、拒绝、恢复、暂停、失败留痕和后续审计语义一致。

## 设计评审检查项
涉及 Agent 能力扩展、缺口修复、tool surface 设计或 prompt 变更时，至少应检查以下问题：

1. 这次改动是在给 Agent 更多上下文与能力，还是在替 Agent 预先做决定。
2. 这次改动是否只对单一站点、单一页面或单一下载链路成立；如果是，默认应视为设计异味。
3. 自然语言约束是否放在 prompt 资产中，而不是散落在 runtime、服务代码或工具描述里。
4. 新增工具是否保持原子能力边界，而不是把工作流伪装成工具。
5. 高风险动作的审批是否由基础设施层保证，而不是靠模型自觉。
6. 新经验是否能通过 skill / memory 机制在未来复用，而不是通过 core 补丁一次性写死。
7. 主 Agent 历史自然语言是否保持在业务级抽象，而不是泄露网页执行细节或工具链路内部状态。
8. 展示链路是否基于结构化来源重建，而不是通过字符串规则去改写 Agent 历史自然语言。

## 变更原则
当实现需要偏离本文档时，必须明确说明偏离的是哪一类边界：
- 智能决策边界
- 通用能力与站点经验边界
- skill 沉淀边界
- 审批治理边界
- 缺口修复优先级边界

任何偏离，都应先更新本文档与相关上位规范，再修改实现，而不是让代码先行漂移。
