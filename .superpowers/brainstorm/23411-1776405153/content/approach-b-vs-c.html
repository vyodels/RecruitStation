Loop Agent 架构方案对比

左边是方案 B（先拉出 Loop Agent），右边是方案 C（先抽统一 Agent OS）。重点看：改动位置、演进方向、当前落地成本。

方案 B：Loop Agent 在上，复用现有 Worker Runtime
结构图
Loop Agent（新）
↓ 派发 worker tasks
现有 Worker Runtime
AgentRun / WorkItem
Queue / Checkpoint
↓ 调用
Skill / Tool
Memory / Policy

优点

  • 最快落地 Loop Agent
  • 保留现有 runtime 资产
  • 风险低，改动集中在 orchestration 层

代价

  • 后面做 Assistant 还要再抽一轮共性
  • 存在一个阶段性的不完全对称结构

适合现在： 先验证“常驻智能 Loop Agent”这个产品形态。

方案 C：先抽 Agent OS，再挂 Loop / Assistant
结构图
Agent Kernel（新）
Prompt Exec
Policy
Approvals
Memory
Skill Lifecycle
Ctx Mgmt
↓ 两个顶层 agent 共用
Loop Agent
Assistant Agent
↓ 再连接 runtime / tools
Worker Runtime
Skills / Tools

优点

  • 结构最干净
  • Loop / Assistant 从第一天就对称
  • 更利于长期扩展更多 agent

代价

  • 现在要先重构底层概念
  • 很容易过度平台化
  • 会延缓 Loop Agent 的验证

适合现在： 你已经确定要先做 Agent OS，而不只是先落地 Loop Agent。

我当前推荐
B → C 演进式路线：现在先做 B，把 Loop Agent 做成真正的顶层 agent；设计时把 profile、memory、policy、skill lifecycle 抽成中性接口，后续再自然长成 C。
