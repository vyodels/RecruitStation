C-lite vs 通用 Agent OS：结构和执行过程对比

这页不是只看“长什么样”，还看“怎么跑”。左边是我建议的 C-lite，右边是更激进的通用 Agent OS。重点看：业务表的位置、world state 的来源、一次 loop tick 的执行路径。

一、结构对比
C-lite：最小内核 + 领域外壳
Loop Agent / Assistant Agent
↓ 使用统一内核能力
Policy / Approval
Memory Retrieval
Skill Lifecycle
Agent Kernel（最小）
↓ 通过领域接口访问业务世界
RecruitingWorldStateProvider
RecruitingActionAdapters
Candidate / Application / JD 表
Recruiting Runtime / Queue / Run
Recruiting Memory Schemas

含义：招聘表继续是招聘表；Kernel 不拥有业务世界，只是访问和驱动它。

通用 Agent OS：大一统平台
Loop Agent / Assistant Agent / 其他 Agent
↓ 全都挂统一平台
Universal Agent Kernel
Universal World State
Generic Memory Store
Unified Runtime
Generic Entity Records
Generic Workflow State
Generic Action Bus
Recruiting / Sales / Support / 其他领域都作为插件接入

含义：为了追求平台统一，往往会诱导你把招聘表语义也抽掉，变成通用实体系统。

二、一次 Loop Tick 的时序对比
C-lite 执行时序
Scheduler
  → wake Loop Agent
Loop Agent
  → Kernel: assemble stable context
  → RecruitingWorldStateProvider: query DB snapshot
  → MemoryIndexProvider: list relevant memory indices
  → LLM: decide next actions
  → if need detail:
       MemoryDetailReader: read selected detail
       LLM: refine action
  → RecruitingActionAdapter / WorkerRuntime
       - enqueue run / call skill / write business data
  → EventBus / Approval / SleepPolicy
  → return next wake condition

特点：世界状态直接来自招聘 DB；memory 是两阶段；Loop tick 是独立的，不依赖长消息历史。

通用 Agent OS 执行时序
Scheduler
  → wake Loop Agent
Loop Agent
  → Universal Kernel: fetch generic world state
  → WorldState Normalizer: map recruiting data into universal entities
  → Generic Memory Store: retrieve context
  → LLM: decide next actions
  → Generic Action Bus
       → Recruiting plugin rehydrates business meaning
       → enqueue run / call skill / write data
  → Universal Event / Approval / Sleep
  → return next wake condition

特点：先抽象成通用状态，再还原回招聘语义；理论更统一，但中间多了一层“翻译层”。

三、关键执行差异

C-lite

读世界：直接查招聘表

LLM 看见的是：招聘语义的真实快照

改数据：通过 recruiting action adapter

复杂度来源：领域逻辑本身

通用 Agent OS

读世界：先标准化成通用实体

LLM 看见的是：一层抽象后的通用状态

改数据：通过通用 action bus 再映射回领域

复杂度来源：领域逻辑 + 抽象/翻译层

四、什么时候选哪一个

更适合 C-lite

  • 当前只有招聘领域真正成熟
  • Loop Agent 是最急的验证对象
  • 你明确不想让 agent 平台反过来吞掉业务表语义

更适合通用 Agent OS

  • 已经有 3 个以上成熟领域要统一
  • 你确定平台抽象本身是眼前最重要目标
  • 你愿意先接受更重的平台设计成本
A

倾向 C-lite

统一 Kernel，但世界状态仍走招聘领域外壳

B

倾向通用 Agent OS

接受通用 world state / action bus / plugin 层
