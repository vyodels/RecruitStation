Tick Cycle 补充：Re-enter Loop 调用链路图

重点说明：为什么一个 tick 内可以多轮 assistant → tool → assistant，以及 Re-enter Loop 到底回到哪一层。

1. Re-enter Loop 的位置
Re-enter Loop 只发生在内层 LLM Turn Loop，不会重新开始整个 Tick
外层 Tick Cycle:
Trigger / Wakeup
→ Bootstrap
→ Sense
→ Update Context
→ Assemble Request
→ [进入内层 LLM Turn Loop]
→ Evaluate
→ Update Memory
→ Continue / Sleep / Respond / Escalate

内层 LLM Turn Loop:
Call LLM
→ Assistant Turn
→ Guard / Policy Check
→ Act
→ Observe
→ Re-enter Loop
    ↳ 回到 Call LLM（不是回到 Trigger / Bootstrap）

关键点：Re-enter Loop 只是在同一个 tick 内，把新的 tool result 再喂给 LLM 继续推理，不代表重新开始一个新 tick。

2. 调用链路图
Assistant → Tool → Observe → Re-enter Loop
Agent Assembly
  ↓ 提供 prompt/context/tool/memory/policy
Agent Kernel
  ↓
Assemble Request
  ↓
Call LLM
  ↓
Assistant Turn
  ├─ 情况 A：直接给 Final Result
  │    → 退出内层 loop
  │
  └─ 情况 B：发起 Tool Call
       ↓
    Guard / Policy Check
       ├─ 不允许 / 需审批
       │    → 返回审批事件 / 阻断结果
       │    → 退出内层 loop
       │
       └─ 允许执行
            ↓
         Act
            ↓
         Tool / Skill / Worker
            ↓
         Observe
            ↓
         生成 tool_result / side effect summary / error
            ↓
         Append result into current tick context
            ↓
         Re-enter Loop
            ↓
         Call LLM（同一个 tick 内的下一轮）
3. Re-enter Loop 时，到底新增了什么

不会重做的

不会重新 Trigger / Wakeup

不会重新 Bootstrap 整个 Assembly

不会重新完整 Sense 世界状态

会新增的

新增一个 tool_result

新增 side effect summary / error

必要时补充少量新的 context detail

然后回到哪

回到 Call LLM

让 assistant 基于新增结果继续决策

4. 一个具体例子
例：Loop Agent 想同步候选人简历
Call LLM
→ assistant: "调用 sync_resume(candidate_id=123)"
→ Guard: 权限允许
→ Act: 执行 sync_resume
→ Observe: 返回 error = "resume missing"
→ Append tool_result: {status:error, reason:"resume missing"}
→ Re-enter Loop
→ Call LLM again
→ assistant: "先调用 parse_resume(candidate_id=123)"
→ Guard
→ Act
→ Observe: parse 成功，得到 resume artifact
→ Re-enter Loop
→ Call LLM again
→ assistant: "再次调用 sync_resume(candidate_id=123)"
→ ...
→ 最终 assistant 输出："同步完成"

含义：这个过程都发生在同一个 tick 内，只是 LLM 经过了多轮工具反馈才拿到最终结果。
