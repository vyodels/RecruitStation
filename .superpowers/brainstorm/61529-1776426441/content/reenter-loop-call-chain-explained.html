Re-enter Loop 调用链路图（带每一步说明）

这页专门把“Assistant → Tool → Observe → Re-enter Loop”的链路拆开，并给每一步补上文字说明，方便你直接顺着看。

每一步说明

1. Agent Assembly

这一步提供这次执行所需的装配配置，不直接执行逻辑。

主要包括：prompt profile、context profile、tool profile、memory profile、policy profile、wakeup/session profile。

2. Agent Kernel

统一执行底座开始工作，负责后续的上下文组装、LLM 调用、工具执行、审批判断、结果回流等流程控制。

3. Assemble Request

把这次需要发给 LLM 的信息组装出来。

典型内容包括：system prompt、动态 user context、可用 tools、memory index，以及必要时选中的 memory detail。

4. Call LLM

真正发起一次模型调用。

此时会把组装好的 system / user / tools 一起发给 LLM，请它给出当前轮的决策。

5. Assistant Turn

接收模型当前轮输出。

这里会有两种结果：一种是直接给出 final result；另一种是要求调用某个 tool / skill / worker。

6A. Final Result

如果 assistant 已经得到足够信息，直接产出最终结果，那么内层 loop 结束，回到外层 tick 的 Evaluate / Update Memory / 出口阶段。

6B. Tool Call

如果 assistant 认为还需要进一步行动或获取信息，就会发起 tool call。

这只是“提出动作请求”，还没有真正执行。

7. Guard / Policy Check

在真正执行 tool 之前，对动作做守卫和策略判断。

这里会检查：权限是否允许、是否需要审批、是否超预算、是否重复执行、是否有高风险副作用。

8A. 不允许 / 需审批

如果动作不允许直接执行，或需要审批，就不会继续进入 Act。

系统会返回一个审批事件、阻断结果或等待状态，然后退出当前内层 loop。

8B. 允许执行

如果策略检查通过，才会真正进入执行阶段。

9. Act

执行 assistant 决定的动作。

可能是调用一个 tool、触发一个 skill、调用一个 worker，或者直接写业务数据。

10. Tool / Skill / Worker

这是动作的真正执行体。

Tool 偏轻量直接能力，Skill 偏封装后的能力资产，Worker 偏较重的后台执行单元。

11. Observe

读取执行结果，观察动作的真实反馈。

结果可能是成功、失败、部分成功、异常、产物生成、状态变化等。

12. 生成 tool_result / summary / error

把执行结果标准化成当前 tick 可继续使用的事实。

例如：tool_result、side effect summary、error、artifact output、status transition。

13. Append result into current tick context

把刚才得到的结果追加回这一次 tick 的上下文中。

注意：这里只更新“当前 tick 上下文”，不会重新开始一个新 tick。

14. Re-enter Loop

表示：在同一个 tick 内，再次回到 LLM 推理。

本质上是“带着新的 tool_result 再问一次 LLM：下一步怎么办”。

15. Call LLM（下一轮）

重新调用 LLM，但不是从零开始。

它能看到刚刚新增的 tool_result，因此可以调整计划、换工具、补动作、或直接结束。

一个关键结论
Re-enter Loop 不等于重新开始一个新 tick。
它只表示：在当前 tick 内，完成了一次 assistant → tool → observe 之后，把新结果回流给 LLM，再继续当前这轮执行。
