Re-enter Loop 调用链路图（框图箭头版）

这版用框图和回跳箭头显示 Re-enter Loop 到底回到哪里。重点看右侧的回流箭头：它回到 Call LLM，而不是重新开始整个 tick。

主链路
Assistant → Tool → Observe → Re-enter Loop
Agent Assembly — 提供这次执行所需的装配配置：prompt、context、tool、memory、policy、wakeup/session profiles
↓
Agent Kernel — 统一执行底座开始工作，负责上下文组装、LLM 调用、工具执行、审批判断、结果回流
↓
Assemble Request — 组装这次发给 LLM 的完整请求：system prompt、dynamic user context、tools、memory index，必要时补 memory detail
↓
Call LLM — 发起一次真实的模型调用，把当前 tick 内已经准备好的信息交给 LLM
↓
Assistant Turn — 接收模型输出；可能直接给 Final Result，也可能发起 Tool Call
路径 A：直接结束
Final Result — 模型已经拿到足够信息，直接输出最终结果
↓
Exit Inner Loop — 退出内层 LLM loop，回到外层 tick 的 Evaluate / Update Memory / 出口阶段
路径 B：Tool Call 后 Re-enter Loop
Tool Call — 模型要求进一步行动，例如调 tool、skill、worker，或继续获取信息
↓
Guard / Policy Check — 检查权限、审批、预算、重复执行、高风险副作用
↓
Act — 执行动作：调用 tool、skill、worker，或修改业务数据
↓
Observe — 读取执行结果，拿到成功 / 失败 / 部分成功 / 异常 / 状态变化
↓
Append Result — 把 tool_result / error / side-effect summary 追加回当前 tick context
Re-enter Loop — 不重新开始 tick，只是在同一个 tick 内继续下一轮 LLM 推理
↺
Back to Call LLM — 回到当前 tick 内的下一轮 Call LLM
关键结论： Re-enter Loop 的回跳目标是 Call LLM，不是 Trigger / Wakeup，不是 Bootstrap，也不是重新开始一个新 tick。
